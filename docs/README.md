# Docs Structure

All non-code project knowledge lives under `docs/`.

## Folder Rules

- `architecture/`: durable technical reference such as system structure, role mapping, setup notes, and project overview
- `plans/`: design docs, proposals, and implementation plans written before or during delivery
- `implementation/`: durable feature writeups describing what was built and why it changed
- `testing/`: QA notes, audits, cross-checks, and testing evidence
- `laporan/`: stakeholder-facing reports
- `operations/`: internal repo working docs such as workflow rules, implementation tracker, repo memory, and session handoffs

## Placement Rule

Use `docs/` for every markdown artifact that should stay with the repository.

- Put it in `operations/` if the file exists to run the work.
- Put it elsewhere in `docs/` if the file explains the software or its delivery.
